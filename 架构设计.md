# liangz77.cn 轻量服务器架构设计

## 目标

在阿里云轻量应用服务器上部署 `liangz77.cn` 相关站点，满足以下需求：

- 中国大陆用户可以直接访问。
- 主站、多个静态前端站点、多个带后台服务的网站共用一台轻量服务器。
- 每个小产品可以独立部署、独立维护、独立扩展。
- 访问量不大时，尽量保持低成本、低复杂度、低运维负担。
- 避免过早引入 Kubernetes、RDS、微服务网关、复杂 CI/CD 等重型设施。

核心原则：

> 先用简单稳定的单机架构快速上线。等某个产品真的有流量、有收入、有瓶颈，再只为那个产品升级架构。

当前已确定的技术取向：

- 开发原则：简单优先，少抽象，少依赖，少组件。
- 部署方式：首选 `git push` 自动部署，备用本地一键部署脚本。
- 发布策略：尽量渐进发布，先上传新版本，再切换和重启，避免粗暴覆盖。
- Web 入口：使用 Caddy，不使用 Nginx。

## 方法论

这套架构参考“低成本独立开发者技术栈”的思路：

- 用一台便宜、稳定的 VPS/轻量服务器承载多个小产品。
- 用 Go 编译单文件二进制，降低部署和运行成本。
- 用 SQLite + WAL 作为默认数据库，避免一开始就上 RDS/PostgreSQL。
- 用 systemd 管理进程，避免引入 Docker/Kubernetes。
- 用 Caddy 作为统一 HTTPS 入口和反向代理。
- 静态前端构建后直接作为文件托管，不让 Node 服务长期常驻。
- 用自动备份保护数据，而不是一开始购买复杂高可用服务。

## 简单开发原则

所有项目默认遵循以下原则：

- 能用一个进程解决，就不拆多个服务。
- 能用 SQLite 解决，就不先上 PostgreSQL/RDS。
- 能用静态文件解决，就不跑 Node 服务。
- 能用 Caddy 配置解决，就不写额外网关代码。
- 能用 systemd 管理，就不引入 Docker/Kubernetes。
- 能用普通文件和目录表达清楚，就不引入复杂平台。
- 每个项目尽量保持独立，避免多个小项目共享复杂运行时。
- 先做直接可用的版本，再根据真实使用情况抽象和优化。

代码层面建议：

- 后端优先使用 Go 单体应用。
- 一个项目一个代码仓库或一个清晰子目录。
- 一个项目一个配置文件。
- 一个项目一个 SQLite 数据库。
- 一个项目一个 systemd service。
- 一个项目一个部署脚本。
- 公共逻辑只有在重复出现三次以上，并且确实稳定后再抽取。

这个原则的目标不是追求“技术极简主义”，而是降低长期维护成本，让每个小产品都能快速上线、独立运行、独立替换。

## 本地开发方式

本地开发不需要模拟线上完整环境，按项目类型使用最直接的开发命令。

纯前端项目：

```bash
npm install
npm run dev
```

常见访问地址：

```text
http://localhost:5173
```

Go 后端项目：

```bash
go run .
```

如果后端带前端开发服务器，可以本地分开启动：

```text
前端：npm run dev
后端：go run .
```

前端开发服务器通过环境变量指向后端：

```text
VITE_API_BASE=http://localhost:9001
```

本地开发原则：

- 本地用 `npm run dev`、`go run .` 这类原生命令即可。
- 本地不需要跑 Caddy，除非要测试多子域名或 HTTPS 行为。
- 本地不需要跑 systemd。
- 本地可以直接使用 SQLite 文件，例如 `dev.db`。
- 线上使用 `app.db`，本地和线上数据库分开。
- 本地开发完成后，通过 `git push prod main` 自动部署上线。

如果某个项目需要同时模拟线上路由，例如 `/api` 反向代理，可以优先使用 Vite dev server proxy，而不是在本地引入 Caddy。

## 总体架构

```text
用户浏览器
   |
   | HTTPS
   v
阿里云 DNS
   |
   v
阿里云轻量应用服务器
   |
   v
Caddy 统一入口
   |
   |-- liangz77.cn              -> 主站静态文件
   |-- www.liangz77.cn          -> 主站静态文件或跳转到根域名
   |-- tools.liangz77.cn        -> 静态前端项目 A
   |-- demo.liangz77.cn         -> 静态前端项目 B
   |-- app1.liangz77.cn         -> 127.0.0.1:9001 后端服务
   |-- app2.liangz77.cn         -> 127.0.0.1:9002 后端服务
   |-- api.app2.liangz77.cn     -> 127.0.0.1:9003 后端服务
```

服务器公网只开放：

```text
22    SSH
80    HTTP，用于证书签发和跳转 HTTPS
443   HTTPS
```

所有后端服务只监听本机：

```text
127.0.0.1:9001
127.0.0.1:9002
127.0.0.1:9003
```

外部用户不能直接访问应用端口，只能通过 Caddy 入口访问。

## 推荐技术栈

第一阶段推荐：

```text
服务器：阿里云轻量应用服务器，中国大陆地域
系统：Ubuntu 24.04 LTS
入口：Caddy
后端：Go 优先
数据库：SQLite + WAL 优先
进程管理：systemd
部署方式：首选 git push 自动部署，备用 deploy.ps1 手动部署
备份：定时备份到阿里云 OSS
监控：systemd logs + 简单健康检查 + 阿里云云监控
```

统一使用 Caddy，原因是：

- 配置简洁。
- 自动申请和续期 HTTPS 证书。
- 很适合多个小站点和子域名的反向代理场景。
- 不需要额外维护 certbot。
- 多站点配置更直观。

## 服务器规格建议

起步规格：

```text
最低可用：2 核 2G
更舒服：2 核 4G
系统盘：40G 起
带宽：3M-5M 起步
```

如果只是个人站点、工具站、低访问量后台系统，2 核 2G 已经可以启动。

如果会同时跑多个 Go 后端服务、数据库备份、日志压缩、前端文件托管，2 核 4G 更舒服。

## 目录规划

服务器上统一使用 `/srv` 存放站点和应用：

```text
/srv/
  sites/
    liangz77.cn/
      releases/
      current -> releases/当前版本
      previous -> releases/上一版本
    tools.liangz77.cn/
      releases/
      current -> releases/当前版本
      previous -> releases/上一版本
    demo.liangz77.cn/
      releases/
      current -> releases/当前版本
      previous -> releases/上一版本

  apps/
    app1/
      releases/
      current -> releases/当前版本
      previous -> releases/上一版本
      shared/
        app.db
        config.env
        uploads/
    app2/
      releases/
      current -> releases/当前版本
      previous -> releases/上一版本
      shared/
        app.db
        config.env
        uploads/

  backups/
    sqlite/
    uploads/
```

说明：

- `/srv/sites` 存放纯静态前端项目，每个站点用 `current` 指向当前版本。
- `/srv/apps` 存放带后端的服务，每个应用用 `current` 指向当前二进制版本。
- 每个后端项目有自己的二进制版本目录、配置文件、SQLite 数据库和上传目录。
- `shared` 存放跨版本共享的数据，部署新版本时不能覆盖。
- `/srv/backups` 存放本地备份，再同步到 OSS。

## 域名规划

建议前期保持清晰、保守：

```text
liangz77.cn                主站 / 个人主页 / 导航页
www.liangz77.cn            跳转到 liangz77.cn

tools.liangz77.cn          工具集合
blog.liangz77.cn           博客，如果需要
lab.liangz77.cn            实验项目集合

xxx.liangz77.cn            独立产品前台
api.xxx.liangz77.cn        独立产品 API，可选
admin.xxx.liangz77.cn      后台管理，可选
```

默认建议：

```text
一个产品一个子域名
API 走同域名 /api
后台走同域名 /admin
```

例如：

```text
todo.liangz77.cn
todo.liangz77.cn/api
todo.liangz77.cn/admin
```

这样可以减少跨域问题，也减少 DNS 和证书管理复杂度。

只有在确实需要前后端完全分离时，再拆成：

```text
todo.liangz77.cn
api.todo.liangz77.cn
```

## DNS 解析

在阿里云 DNS 中配置：

```text
@       A      服务器公网 IP
www     CNAME  liangz77.cn
tools   A      服务器公网 IP
demo    A      服务器公网 IP
app1    A      服务器公网 IP
app2    A      服务器公网 IP
```

也可以配置泛解析：

```text
*       A      服务器公网 IP
```

但前期不建议使用泛解析。明确一个子域名配一个记录，更清楚，也更方便排错和备案管理。

## Caddy 配置示例

纯静态站点：

```caddyfile
liangz77.cn, www.liangz77.cn {
    root * /srv/sites/liangz77.cn/current
    file_server
}

tools.liangz77.cn {
    root * /srv/sites/tools.liangz77.cn/current
    file_server
}

demo.liangz77.cn {
    root * /srv/sites/demo.liangz77.cn/current
    file_server
}
```

独立后端服务：

```caddyfile
app1.liangz77.cn {
    reverse_proxy 127.0.0.1:9001
}

app2.liangz77.cn {
    reverse_proxy 127.0.0.1:9002
}
```

前端静态文件 + API 后端：

```caddyfile
todo.liangz77.cn {
    handle /api/* {
        reverse_proxy 127.0.0.1:9001
    }

    handle /admin/* {
        reverse_proxy 127.0.0.1:9001
    }

    handle {
        root * /srv/sites/todo.liangz77.cn/current
        file_server
        try_files {path} /index.html
    }
}
```

`try_files {path} /index.html` 用于支持 React/Vue/Svelte 等前端路由。

这里 `root` 明确指向 `current`，而不是站点父目录。这样发布新版本时，只需要切换 `current` 软链接，Caddy 就会无感知地服务新版本内容。

## 后端服务形态

每个后端项目建议编译为一个 Go 单文件二进制。

为了支持渐进发布，应用目录分为 `releases`、`current`、`previous` 和 `shared`。

```text
/srv/apps/todo/
  releases/
    20260417-153000/
      todo
    20260418-101500/
      todo
  current -> releases/20260418-101500
  previous -> releases/20260417-153000
  shared/
    app.db
    config.env
    uploads/
```

每个项目：

- 一个独立目录。
- 一个独立端口。
- 一个独立 systemd service。
- 一个独立 SQLite 数据库。
- 一个独立上传目录。

示例 systemd service：

```ini
[Unit]
Description=todo service
After=network.target

[Service]
WorkingDirectory=/srv/apps/todo/shared
EnvironmentFile=/srv/apps/todo/shared/config.env
ExecStart=/srv/apps/todo/current/todo
Restart=always
RestartSec=3
User=www-data
Group=www-data

[Install]
WantedBy=multi-user.target
```

手工部署流程示例：

```bash
go build -o todo
scp todo server:/srv/apps/todo/releases/20260418-101500/todo
ssh server
old_target=$(readlink -f /srv/apps/todo/current)
ln -sfn "$old_target" /srv/apps/todo/previous
ln -sfn /srv/apps/todo/releases/20260418-101500 /srv/apps/todo/current
systemctl restart todo
```

实际使用时不直接手工执行这些命令，而是由 `git push` 触发服务器上的部署脚本自动执行。

注意：`previous` 必须指向“旧的 release 实际目录”，不能指向 `current` 这个符号链接本身。否则后续 `current` 被切走后，`previous` 也会一起漂移，回滚会失效。

## Git Push 自动部署

默认部署方式采用自托管 Git 裸仓库 + `post-receive` hook。

目标体验：

```powershell
git add .
git commit -m "update"
git push prod main
```

代码推送到服务器后，服务器自动完成构建、发布、重启和健康检查。

整体链路：

```text
本地开发机
  |
  | git push prod main
  v
阿里云轻量服务器上的 bare git repo
  |
  | post-receive hook
  v
服务器自动部署脚本
  |
  | 构建 / 切换版本 / 重启 / 健康检查
  v
正式上线
```

这里的 `prod` 不是 GitHub，而是你的服务器：

```text
prod = ssh://deploy@服务器公网 IP/srv/git/todo.git
```

本地配置 remote 示例：

```bash
git remote add prod deploy@服务器公网IP:/srv/git/todo.git
```

之后日常发布只需要：

```bash
git push prod main
```

### 为什么不用 GitHub 作为第一版

第一版不依赖 GitHub Actions 或第三方 CI，原因是：

- 代码可以不放到第三方平台。
- 国内访问链路更短。
- 不需要维护 GitHub Secrets。
- 不需要单独配置 Runner。
- 不需要额外 CI/CD 平台。
- 服务器收到代码后可以直接部署自己，链路最短。

以后如果需要多人协作、代码审查、云端 CI，再接入 GitHub、Gitea 或 GitLab。

### post-receive hook 流程

服务器上的 `post-receive` hook 做这些事：

```text
1. 接收 git push
2. checkout 最新代码到 /srv/build/项目名
3. 执行项目构建命令
4. 创建 releases/时间戳 目录
5. 复制构建产物到新 release
6. 读取旧 current 的真实目录，记录为 old_target
7. 用 old_target 更新 previous
8. 切换 current 指向新 release
9. 重启对应 systemd 服务，静态站点则不需要重启
10. 调用 /healthz 做健康检查
11. 成功则完成发布
12. 失败则切回 previous 并重启
```

建议把部署脚本中的切换动作当作固定协议，而不是“差不多这样做”：

```text
old_target = readlink -f current
previous   = old_target
current    = new_release
restart service
health check
if failed:
  current = previous
  restart service
```

这样回滚依赖的是确定的 release 目录，而不是某个仍会变化的软链接。

### 静态前端自动部署

适用于 Vite/React/Vue/Svelte 等项目：

```text
git push prod main
  -> 服务器 checkout 代码
  -> npm ci
  -> npm run build
  -> 复制 dist 到 releases/时间戳
  -> previous 指向旧 current 的真实目录
  -> 切换 current
  -> Caddy 自动服务新版本
```

Caddy 始终指向：

```text
/srv/sites/tools.liangz77.cn/current
```

### Go 后端自动部署

适用于 Go 后端项目：

```text
git push prod main
  -> 服务器 checkout 代码
  -> go build -o todo
  -> 复制二进制到 releases/时间戳
  -> previous 指向旧 current 的真实目录
  -> 切换 current
  -> systemctl restart todo
  -> curl http://127.0.0.1:9001/healthz
  -> 失败则回滚
```

### 部署用户与权限边界

建议把“接收代码/构建”和“运行服务”分成两层权限：

```text
deploy 用户：
- 接收 git push
- 执行 post-receive hook
- 写 /srv/build、/srv/sites/*、/srv/apps/*/releases
- 更新 current、previous 软链接

运行用户，例如 www-data 或 app-todo：
- 运行 systemd 服务
- 只读 current 版本目录
- 读写 /srv/apps/项目/shared
```

推荐额外约束：

```text
deploy 用户不要直接拥有 shared 下业务数据的完全权限
deploy 用户不要拥有任意 sudo
只放行部署所需的 systemctl restart/status 权限
config.env 仅 root 和运行用户可读
```

如果使用 `deploy` 用户触发发布，推荐通过 sudoers 只开放白名单命令，例如：

```text
deploy ALL=NOPASSWD: /bin/systemctl restart todo, /bin/systemctl status todo
```

这样既能自动发布，又不会把整台服务器的 root 权限暴露给 Git hook。

## 一键部署与渐进更新

本节作为备用部署方案。默认优先使用 `git push` 自动部署。

如果暂时没有配置 Git 裸仓库，也可以先用本地一键部署脚本。

部署目标：

```text
本地改代码
本地构建
一条命令部署到服务器
服务器先接收新版本
通过基本检查后再切换
重启对应服务
保留上一个版本，方便回滚
```

建议每个项目都有自己的部署脚本：

```text
deploy.ps1     Windows 本地开发环境使用
deploy.sh      macOS/Linux 本地开发环境使用，可选
```

你的当前开发环境是 Windows，因此第一优先级是 `deploy.ps1`。

### 静态前端部署流程

例如 Vite/React/Vue/Svelte 项目：

```text
1. 本地执行 npm run build
2. 生成 dist/
3. 上传到服务器 releases/新时间戳/
4. 记录旧 current 的真实目录到 previous
5. 在服务器上切换 current 指向新版本
6. 如需验证静态资源可用，执行 HTTP 检查
7. 失败则切回 previous
```

推荐服务器目录：

```text
/srv/sites/tools.liangz77.cn/
  releases/
    20260417-153000/
    20260418-101500/
  current -> releases/20260418-101500
  previous -> releases/20260417-153000
```

Caddy 指向：

```text
/srv/sites/tools.liangz77.cn/current
```

这样部署新版本时，不直接覆盖当前线上目录，而是新增一个 release，再切换软链接。

### Go 后端部署流程

部署流程：

```text
1. 本地 go build
2. 上传新二进制到 releases/新时间戳/
3. 服务器执行 chmod +x
4. 可选：执行 todo --version 或 todo healthcheck
5. 读取旧 current 的真实目录并更新 previous
6. 切换 current 到新 release
7. systemctl restart todo
8. 检查 /healthz
9. 如果失败，切回 previous 并重启
```

### 一键部署命令目标

最终希望本地只需要执行：

```powershell
.\deploy.ps1
```

或带环境：

```powershell
.\deploy.ps1 -Env prod
```

部署脚本应该做这些事：

```text
检查本地工作区状态
构建项目
生成 release 时间戳
上传文件到服务器
记录旧 current 的真实路径
更新 previous
在服务器上切换 current
重启 systemd 服务
检查健康接口
输出部署结果
```

如果健康检查失败：

```text
切回 previous
重启服务
输出失败原因
```

### 回滚命令

每个项目后续也可以提供：

```powershell
.\deploy.ps1 -Rollback
```

它只做：

```text
读取 previous 的真实目录
将 current 切回 previous 指向的 release
重启服务
检查 /healthz
```

第一版就应该把回滚逻辑一起实现。部署可以简化，但回滚不能留到以后，否则第一次线上异常时就没有兜底。

## SQLite 策略

每个应用使用一个独立 SQLite 数据库：

```text
/srv/apps/todo/app.db
/srv/apps/notes/app.db
/srv/apps/shop/app.db
```

应用启动时建议设置：

```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
PRAGMA foreign_keys=ON;
PRAGMA busy_timeout=5000;
```

适合 SQLite 的场景：

- 个人工具。
- 小型 SaaS。
- 后台管理系统。
- 内容站。
- 表单系统。
- 低并发产品。
- 读多写少系统。

暂时不适合 SQLite 的场景：

- 高频并发写入。
- 复杂队列系统。
- 大量多人实时协作。
- 强分析型查询。
- 数据库需要多机器共享写入。

如果某个产品真的增长起来，再单独把那个产品迁到 PostgreSQL 或 RDS。

## 静态前端项目

前端项目构建后直接部署为静态文件。

例如 Vite 项目：

```bash
npm run build
```

生成：

```text
dist/
```

部署到：

```text
/srv/sites/tools.liangz77.cn/releases/新时间戳/
```

原则：

- 前端项目构建后就是静态文件。
- 不要为了静态站点长期运行 Node 服务。
- Caddy 直接托管静态文件。
- Caddy 指向 `current`，部署时切换软链接，不直接覆盖线上目录。

## 带后台的网站

后台项目可以按复杂度分三类。

轻量后台：

```text
前台 + 后台 + API 都在同一个 Go 服务里
```

中等复杂：

```text
前端静态文件由 Caddy 托管
API 和后台由 Go 服务提供
```

复杂项目：

```text
单独工程
单独端口
单独 SQLite
单独 systemd service
```

推荐默认形态：

```text
crm.liangz77.cn
  /             前端页面
  /api          API
  /admin        后台
```

对应架构：

```text
Caddy -> 静态文件 + reverse_proxy /api /admin -> Go 服务
```

## 备案

因为服务器计划放在阿里云中国大陆地域，并且面向中国大陆用户访问，需要注意备案。

通常流程：

```text
1. 域名 liangz77.cn 完成实名认证
2. 购买中国大陆地域的阿里云轻量服务器
3. 在阿里云备案系统提交 ICP 备案
4. 备案通过后，域名解析到服务器
5. 网站底部放 ICP 备案号
6. 开站后按要求做公安联网备案
```

注意：

- 中国大陆地域服务器对外提供网站服务前，通常需要完成 ICP 备案。
- 备案主体、网站内容、接入服务器信息要保持一致。
- 子域名一般依附于已备案主域名使用，但如果网站内容、主体或接入信息变化，应按阿里云备案规则处理。
- 备案和公安联网备案不要省略，否则后续可能影响访问。

## 备份策略

这个架构最大的风险不是算力，而是数据丢失。

建议：

```text
每天凌晨备份 SQLite
每天凌晨备份 uploads
保留最近 7 天本地备份
同步一份到阿里云 OSS
每周做一次完整归档
```

SQLite 备份不要只粗暴复制正在写入的 `app.db` 文件。

最低限度需要一起备份：

```text
app.db
app.db-wal
app.db-shm
```

更稳的方式是使用 SQLite 官方备份机制，或者由应用提供备份命令：

```bash
todo backup /srv/backups/sqlite/todo-2026-04-17.db
```

备份目标：

```text
本地：/srv/backups
远端：阿里云 OSS
```

## 安全基线

服务器至少应做到：

```text
SSH 禁止密码登录，使用密钥
服务器防火墙只开放 22、80、443
应用端口只监听 127.0.0.1
每个服务使用低权限用户运行
后台管理必须有登录
重要后台可以额外加 Basic Auth 或 IP 白名单
定期 apt update
日志不要打印密码、token、cookie
config.env 权限收紧
deploy 用户和运行用户分离
Git hook 不直接拿 root 常驻权限
```

后台站点建议：

```text
应用自己的登录
+ Caddy Basic Auth
```

小访问量后台系统使用双层保护，成本很低，但安全性提升明显。

## 监控和日志

第一阶段不建议上 Prometheus/Grafana。

先使用：

```text
systemctl status app1
journalctl -u app1 -f
caddy logs
阿里云云监控
定时 curl 健康检查
```

每个 Go 服务提供：

```text
/healthz
```

返回：

```json
{"ok":true}
```

服务器可以用 cron 每分钟检查关键服务健康状态。失败时先记录日志，后续可以接入邮件、企业微信、钉钉等通知方式。

## 升级路径

这套架构可以先撑很久。只有出现以下情况，才考虑升级：

```text
某个服务 CPU 长期满载
SQLite 写锁成为瓶颈
图片/文件流量明显变大
服务器磁盘快满
单机故障已经不能接受
某个产品开始有真实收入，需要更高 SLA
```

升级原则：

> 不要全局升级，只升级那个变大的产品。

升级路径：

```text
静态资源多 -> OSS + CDN
数据库写入多 -> PostgreSQL / RDS
单服务压力大 -> 单独 ECS
可用性要求高 -> 多机 + 负载均衡
后台任务多 -> 加队列
```

`liangz77.cn` 这台轻量服务器长期作为“产品孵化母机”。
