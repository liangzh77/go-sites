# 站点迁移审计

## 目标

评估首页以下三个栏目中的外链页面，判断是否适合迁入当前项目并改为站内链接：

- `实用工具`
- `娱乐与教育`
- `张牧作品`

目标不是简单镜像，而是以“代码整洁、结构合理、清晰、易于维护”为前提，分批纳入当前仓库。

## 当前架构约束

- 当前线上站点是纯静态站点，由 `Caddy` 直接托管文件。
- 当前仓库没有前端构建工具链，也没有多应用目录结构。
- 如果要迁入多个纯前端页面，建议按“多静态子应用”模式组织，而不是把所有页面平铺在根目录。

## 调查方法

- 读取当前首页数据源 [data.js](C:/data/liang/code/github_projects/go-sites/data.js) 中三个栏目的外链列表。
- 对代表性目标站点做域名和托管平台识别。
- 结合页面名称、功能描述、托管平台类型，判断其是否大概率为纯静态前端页。

说明：

- `card910.netlify.app`、`snack-snake.netlify.app`、`tankbattle-warriors.netlify.app` 解析结果明确落在 `Netlify`。
- `wallet-liangz77.vercel.app` 解析结果明确落在 `Vercel`。
- 当前环境对部分外站 HTTPS 握手不稳定，因此下面部分判断属于“基于托管平台 + 页面功能描述”的工程推断，不等于最终源码级确认。

## 迁移分级

### B. 需要验证后再迁移

这批页面“可能是纯前端”，但更容易隐藏后端依赖、第三方接口或构建路径问题。

#### 实用工具

- `时间线` `https://timeline-book.vercel.app`
- `投资记录` `https://invest-record.vercel.app`
- `家庭记账本` `https://wallet-liangz77.vercel.app`

风险点：

- 可能使用 `localStorage`，也可能接了远端接口。
- `Vercel` 项目有概率使用 `API Routes`、中间件或构建期路由。
- 这类数据型工具即使无后端，也需要额外确认导入导出、路由和资源引用。

#### 娱乐与教育

- `你画AI猜` `https://draw-guess-lime.vercel.app`
- `旅游攻略` `https://travel-draft.vercel.app`

风险点：

- `你画AI猜` 名称里有 `AI`，可能调用第三方推理接口。
- `旅游攻略` 可能是纯静态内容站，也可能含动态搜索、地图或远端数据源。

#### 张牧作品

- `Woolly Run` `https://woolly-run.vercel.app`
- `触角公司网站` `https://chujiao.vercel.app`

风险点：

- `Woolly Run` 可能仍是纯前端，但如果带资源懒加载或特定构建输出，需要核对。
- `触角公司网站` 可能是多页静态站，也可能有表单、国际化、构建产物路径和运行时配置。

### C. 暂不建议盲迁

当前三个栏目里，没有明显必须放弃的目标，但以下情况一旦出现，就不应继续按“静态页收编”路线做：

- 站点依赖服务端接口或 `Vercel/Netlify Functions`
- 站点只有构建产物，没有可维护源码
- 站点资源路径硬编码原域名，改成本域路径后修改量过大
- 站点依赖第三方私有 API key、鉴权或回调域名

## 推荐接入方式

不建议把所有页面直接塞进仓库根目录。建议改为多应用静态结构：

```text
/
  index.html
  data.js
  apps/
    tools/
      card910/
      image-stitch/
      pic-wall/
      list-chore/
    fun/
      snack-snake/
      digital-maze/
      tetris888/
      match-3-game/
      jump100/
      whoswolf/
      apollonius/
      fermat-point/
      parabolic-equation/
      pascal-brianchon/
      poly-deriv/
    works/
      tankbattle-warriors/
      tank-1v1/
      starcruiser/
      towerfun/
  shared/
    assets/
    styles/
    scripts/
  scripts/
    audit/
```

说明：

- `apps/` 下每个页面目录都应该是一个独立静态子应用。
- 公共资源再抽到 `shared/`，不要在一开始就强行共用所有代码。
- 首页只负责导航，不要把每个子站逻辑混进根目录。

## 推荐 URL 方案

统一改成站内路径，不使用新的域名：

- `/tools/card910/`
- `/tools/image-stitch/`
- `/fun/snack-snake/`
- `/fun/digital-maze/`
- `/works/tankbattle-warriors/`

这样比继续拆二级域名更容易维护，也更适合当前 `Caddy` 和发布脚本。

## 下一步建议

下一步不该直接改首页链接，而应该先做这两件事：

1. 先为第一批页面建立目标目录和命名规则。
2. 再逐个拉取页面资源，确认是否存在：
   - 绝对路径
   - 运行时接口
   - Service Worker
   - 原域名硬编码
   - 构建产物依赖

只有在第一批通过后，才应该回头修改首页的外链为站内链接。
